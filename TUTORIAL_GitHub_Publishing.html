Publishing a napari Plugin on GitHub

A complete, beginner-friendly walkthrough using the Cryo3D Editor plugin as the running example — from understanding what the plugin is, to installing it, to publishing and updating it on GitHub.

Learning objectives

By the end of this tutorial, you will be able to:

  • Explain what a napari plugin is and what the Cryo3D Editor does.
  • Read and understand the file structure of an installable Python plugin (the npe2 layout).
  • Install the plugin on your own machine or on the cluster.
  • Publish the plugin to GitHub using both GitHub Desktop and the terminal.
  • Update the plugin and push new changes safely.
  • Diagnose the most common errors (lock file, authentication, "no matching distribution").

1 · What is the plugin? (the big picture)

Why this exists

napari is a free, open-source image viewer for scientific data, especially 2D/3D/n-D microscopy and cryo-electron tomography (cryo-ET). On its own, napari shows your data and lets you draw labels — but it does not know anything about your specific analysis workflow. A plugin is a small piece of software that "plugs into" napari and adds new buttons, panels, and operations tailored to a task.

The Cryo3D Editor is such a plugin. It adds a dockable panel to napari with five tabs, each solving one cryo-ET cleanup/visualization problem:

TabWhat it doesTypical use
🧹 DustHides or removes small objects, and shrinks objects (per-object erosion)Cleaning noisy segmentations
✂️ EraseA 2D paint-mask eraser for Labels layersManually deleting wrong regions
🎨 ColorsPer-label colour, opacity, merging, presetsMaking structures readable
💡 RenderLighting, contrast, membrane enhancement, snapshot, Save as MRCPublication-quality views & export
🎬 MovieKeyframe 3D animation + 360° spin movieTalks, papers, outreach
🔵 Key concept — two ways to run the same code

The exact same Python file can be used in two modes:

(a) Standalone — you run python cryo3d_editor.py and it opens napari with the panel already attached. Great for quick testing.

(b) Installed plugin — you pip install it once, and afterwards it appears inside any napari session under Plugins → Cryo3D Editor. This is what your colleagues and students will use.

What "publishing on GitHub" means

Git is a version-control system: it records the history of your files so you can go back in time, see what changed, and collaborate. GitHub is a website that hosts git repositories online, so others can download, install, and contribute to your code. Publishing = uploading your local project to GitHub so it has a public web address (a URL).

🟣 Note — GitHub is not the same as PyPI

GitHub hosts the source code. PyPI (the Python Package Index) is a separate service that powers pip install <name> by short name. Until you publish to PyPI, pip install napari-cryo3d-editor will fail — you install from GitHub or from your local copy instead. This is covered in Part 3.

2 · How the plugin is built (package structure)

To be installable, a single script is not enough — Python needs a small set of "metadata" files that describe the package: its name, version, dependencies, and how napari should find the widget. The modern napari standard for this is called npe2 (napari plugin engine 2). Here is the full layout of the Cryo3D Editor:

The three files that make it a plugin (not just a script)

FileWhat it declaresWhy it matters
pyproject.tomlPackage name, version, dependencies, and the line [project.entry-points."napari.manifest"]This is what pip reads to install the package and what tells napari "this project is a plugin".
napari.yamlA command pointing at _widget:Cryo3DEditorWidget and a widget contributionThis is how the panel appears in the Plugins menu. The pointer must match a real class in the code.
__init__.pyImports the widget class and sets __version__Makes the folder importable as napari_cryo3d_editor.
🟡 Important — the manifest must point to real code

In napari.yaml the line python_name: napari_cryo3d_editor._widget:Cryo3DEditorWidget means: "in the file _widget.py, use the class Cryo3DEditorWidget." If you rename that class, you must update this line, or napari will raise an error when you open the plugin.

3 · Installing the plugin

There are three ways to install, depending on whether the code is on your machine, on GitHub, or on PyPI. The first error many people hit is trying the PyPI way before the package is published there.

MethodCommandWhen to use
A. Local (editable)pip install -e .You have the folder on disk and are actively editing it. Changes take effect immediately — no reinstall.
B. From GitHubpip install "git+https://github.com/lifeviewspace/cryo3d-editor.git"On a fresh machine or the cluster, no manual clone needed.
C. From PyPIpip install napari-cryo3d-editorOnly works after you publish to PyPI (not done yet).

Recommended: method A (editable install)

# 1. activate the environment that has napari
conda activate membrain-env

# 2. go into the project folder
cd ~/Documents/GitHub/Cryo3d-editor

# 3. install it in "editable" mode (the dot means "this folder")
pip install -e .

# 4. launch napari — the plugin is now in the Plugins menu
napari
🟢 Output — how to know it worked

After pip install -e . you should see Successfully installed napari-cryo3d-editor-3.0.0. Then in napari, open the menu Plugins → you should see Cryo3D Editor. Click it and the five-tab panel docks on the right.

🔴 Warning — "No matching distribution found"

If you run pip install napari-cryo3d-editor and get ERROR: No matching distribution found, that is expected: the package is not on PyPI. Use method A or B instead. This is not a bug in the plugin.

🟣 Note — the [gpu] extra

pip install "napari-cryo3d-editor[gpu]" adds optional NVIDIA GPU acceleration (CuPy). It does not work on Apple Silicon Macs and is never required — the plugin always falls back to CPU. Skip it on macOS.

4 · Publishing to GitHub — step by step

Publishing has two halves: a local half (turn your folder into a git repository and record a snapshot) and a remote half (create the repository on GitHub and upload). Figure 2 shows the whole journey.

Path 1 — Using GitHub Desktop (recommended for beginners)

GitHub Desktop is a free app that turns the git commands into buttons and handles login for you.

  1. Add the project. Open GitHub Desktop → File → Add Local Repository → choose your cryo3d-editor folder. If it is not a repository yet, it offers to create one — accept.
  2. Review the changes. The left panel lists every changed file with a checkbox. Leave them all checked.
  3. Write a commit message. In the Summary box, type a short description, e.g. Initial commit or Add Save as MRC export.
  4. Commit. Click Commit to main. This records the snapshot locally.
  5. Publish. Click Publish repository at the top. Choose the name (cryo3d-editor), keep it Public, and confirm. GitHub Desktop creates the repo online and uploads everything in one step.
  6. Future changes. After editing code, the changes reappear in the left panel → commit again → click Push origin.
🟡 Important — do NOT add README/license on GitHub if you already have them locally

When creating the repository (the website form or the Publish dialog), leave "Add README", "Add .gitignore", and "Add license" turned off. You already have these files locally; adding them again on GitHub creates a conflicting history that blocks your first push.

Path 2 — Using the terminal (full control)

# --- LOCAL HALF ---
cd ~/Documents/GitHub/Cryo3d-editor
git init                       # step 1: create the repository
git add -A                     # stage every file
git commit -m "Initial commit" # step 2: record the snapshot
git branch -M main             # name the main branch "main"

# --- create the empty repo on github.com/new first (step 3) ---

# --- REMOTE HALF ---
git remote add origin https://github.com/lifeviewspace/cryo3d-editor.git  # step 4
git push -u origin main        # step 5: upload
🔴 Warning — password login no longer works

If git push asks for a password and you type your GitHub password, you get: Password authentication is not supported. GitHub requires a Personal Access Token (PAT) instead. The simplest fix is to push through GitHub Desktop, which is already logged in. To use the terminal, create a token at github.com/settings/tokens (classic, scope repo) and paste it when asked for the password.

🔴 Warning — "A lock file already exists" / index.lock

If a previous git command was interrupted (or GitHub Desktop and the terminal ran at the same time), you may see Unable to create '.git/index.lock': File exists. Close GitHub Desktop, then remove the stale lock and retry:

rm -f ~/Documents/GitHub/Cryo3d-editor/.git/index.lock
🟣 Note — keep code out of cloud-sync folders

Avoid keeping a git repository inside OneDrive/Dropbox/iCloud. The sync service touches the hidden .git folder at the same time as git and can corrupt it (you will see "Operation not permitted" warnings). Keep repositories in a plain folder like ~/Documents/GitHub/.

5 · Updating the plugin (everyday workflow)

After the first publish, the routine for every future change is short and always the same three actions: edit → commit → push.

  1. Edit the code (e.g. fix a bug in _widget.py). If you installed with pip install -e ., just restart napari to see the change — no reinstall.
  2. Commit the snapshot with a message describing what and why: 
    cd ~/Documents/GitHub/Cryo3d-editor
git add -A
git commit -m "Fix napari 0.7 colormap reset (KeyError: colors)"
  3. Push to GitHub so others get the update: 
    git push
🔵 Key concept — write commit messages for your future self

A good message says what changed and why, e.g. Fix Qt6 compatibility (vispy backend detected via qtpy). Six months later, the history becomes a readable changelog instead of a wall of "update" messages.

Bumping the version

When you make a meaningful release, raise the version number in two places so they stay in sync: pyproject.toml (version = "3.1.0") and src/napari_cryo3d_editor/__init__.py (__version__ = "3.1.0"). Then commit and push.
